# Fleet-managed Elastic Agent installation

Fleet-managed mode is the recommended deployment model when you want Kibana and Fleet to install package assets, render Elastic Agent policy, manage upgrades, and centrally configure AgentGuard / AI Sentinel findings collection.

## Build and install the package

From the package directory:

```bash
elastic-package lint
elastic-package build
```

For a local development stack:

```bash
elastic-package stack up
elastic-package install
```

The built package zip is written under `build/packages/` and can be published to an internal Elastic Package Registry for non-development environments.

## Add the integration to an Agent Policy

1. Open Kibana.
2. Go to **Management → Integrations**.
3. Search for **AgentGuard / AI Sentinel**.
4. Add the integration to an Elastic Agent policy.
5. Confirm the input type is `filestream`.
6. Configure findings paths for the operating systems in that policy.

Default paths include:

- Linux: `/var/log/agentguard/findings.ndjson`, `/var/log/ai-sentinel/findings.ndjson`
- macOS: `/Library/Logs/AgentGuard/findings.ndjson`, `/Library/Logs/AI-Sentinel/findings.ndjson`
- Windows: `C:\ProgramData\AgentGuard\logs\findings.ndjson`, `C:\ProgramData\AI-Sentinel\logs\findings.ndjson`

## Enroll Elastic Agent

Use the Fleet enrollment command generated by Kibana for the selected Agent Policy. Install and enroll Elastic Agent on each host that runs, or has access to, the AgentGuard / AI Sentinel findings file.

## Verify policy rendering

In Fleet, open the enrolled agent and inspect the effective policy. The policy should contain:

- `type: filestream`
- `data_stream.dataset: ai_sentinel.findings`
- `paths` pointing at AgentGuard / AI Sentinel NDJSON files
- `pipeline: logs-ai_sentinel.findings-default`
- tags such as `ai_sentinel`, `agentguard`, `ai_activity`, `mcp_monitoring`, and `endpoint_visibility`

See `_dev/examples/fleet-managed-policy-example.yml` for a documentation-only rendering example.

## Verify data in Kibana

1. Open **Discover**.
2. Use the `logs-ai_sentinel.findings-*` data view, or create it if needed.
3. Filter on `event.dataset: ai_sentinel.findings`.
4. Confirm documents land in `logs-ai_sentinel.findings-default` for the default namespace.

## Verify draft Elastic Security rule references

1. Draft rule KQL is documented in `docs/security-rules.md` and TOML drafts are under `_dev/security_rules_toml/`.
2. Convert drafts to supported Elastic Security rule saved-object JSON before packaging them as integration assets.
3. If you create rules manually in a lab, target `logs-ai_sentinel.findings-*`.
4. Generate a safe synthetic finding and verify the expected rule behavior.

## Permission troubleshooting

- **Linux**: ensure the Elastic Agent service user can read `/var/log/agentguard/findings.ndjson` or `/var/log/ai-sentinel/findings.ndjson` and traverse parent directories.
- **macOS**: verify file permissions under `/Library/Logs/AgentGuard` or `/Library/Logs/AI-Sentinel`; grant service access through your MDM or endpoint management tooling as needed.
- **Windows**: ensure the Elastic Agent service account can read `C:\ProgramData\AgentGuard\logs\findings.ndjson` or `C:\ProgramData\AI-Sentinel\logs\findings.ndjson`.
- **No events**: inspect Elastic Agent logs, verify the scanner is writing NDJSON, and confirm the policy path matches the actual file path.
- **Pipeline failures**: search for `event.kind: pipeline_error` or the tag `ai_sentinel_invalid_json`.
